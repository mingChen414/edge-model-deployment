# AMD Ryzen AI Max+ 395（128GB）本地推理模型部署测试报告

> 结论：在 MS-S1 MAX（AMD Ryzen AI Max+ 395、Radeon 8060S、128GB LPDDR）上，`gpt-oss-120b Q4_K_M` 是当前能力、稳定性、长上下文 decode、工具调用生态和内存余量之间最均衡的本地 Agent 模型；长上下文优先 Vulkan，短上下文可用 ROCm/HIP。若优先考虑速度与模型体积，`Qwen-AgentWorld-35B-A3B UD-Q5_K_M` 和 `Qwen3.6-35B-A3B UD-Q4_K_M` 的 ROCm/HIP 路径更合适。
>
> 本报告只保留本机推理速度、内存、质量/能力证据、兼容性和部署全链路。质量结论目前是定性记录，不是统一测试集上的量化评分；跨模型速度来自不同后端或测试配置，不能直接当作严格排名。

### 资源判定原则

Vulkan 的多个 heap 不是一个可任意拼接的连续池。模型能否全量 offload，必须同时估算模型文件、KV cache、计算图、batch/ubatch、prompt cache、系统和驱动开销。以 `gpt-oss-120b Q4_K_M` 为例，原始记录的单 slot 预算约 77.6GB，不含临时计算图及系统/驱动额外开销。

## 4. gpt-oss-120b Q4_K_M 专项结果

### 4.1 推荐部署参数

以下命令按原始记录整理，已将本机路径和局域网地址脱敏。命令在 Windows `cmd.exe` 中执行；`<LLAMA_CPP_DIR>`、`<MODEL_DIR>` 和 `<LAN_IP>` 需要替换为实际值。

```cmd
cd /d <LLAMA_CPP_DIR>

llama-server.exe ^
  -m <MODEL_DIR>\gpt-oss-120b-Q4_K_M-00001-of-00002.gguf ^
  -ngl 99 ^
  -c 131072 ^
  -np 1 ^
  -fa on ^
  -b 8192 ^
  -ub 4096 ^
  --cache-prompt ^
  --cache-ram 10240 ^
  --ctx-checkpoints 32 ^
  --checkpoint-min-step 256 ^
  --host <LAN_IP> ^
  --port 8083
```

| 参数/资源 | 推荐值或实测 | 作用与边界 |
|---|---:|---|
| `-ngl` | `99` | Vulkan 全量 offload 的原始记录配置 |
| `-c` | `131072` | 单 slot 提供完整 128k 上下文 |
| `-np` | `1` | 单路延迟和 decode 最优；多人测试可另行增加 slot，但不能沿用单 slot 速度结论 |
| `-b / -ub` | `8192 / 4096` | 原始记录中较稳定且较快的 batch/ubatch 组合 |
| `-fa` | `on` | 长上下文必须开启的 Flash Attention 配置 |
| `--cache-prompt` | 开启 | 复用相同 token 前缀的 KV，减少重复 prompt eval |
| `--cache-reuse` | 不启用 | 当前 gpt-oss/Vulkan 上下文不支持 KV shifting，日志会提示不支持并自动禁用 |
| `--cache-ram` | `10240` MiB | prompt cache 预算约 10GB，约可容纳两个最长上下文级别缓存 |
| `--ctx-checkpoints / --checkpoint-min-step` | `32 / 256` | 保存中间 KV 状态，便于从已有前缀继续处理 |
| 模型 GGUF | 约 62.8GB | 两片模型必须从 `00001-of-00002` 加载 |
| 128k KV cache | 约 4.8GB | 估算值；实际还会叠加计算图、系统和驱动开销 |
| 总内存预算 | 约 77.6GB | 不含临时计算图及系统/驱动额外开销 |
| 启动到监听 | 约 128s | `-np 1` 原始实测 |

### 4.2 推荐参数下的速度

原始记录标识为 llama.cpp b9754 Vulkan，`-ngl 99 -fa on -t 16 -r 3`。这组数据与第 3 节的跨模型汇总表不是同一已完整描述的测试口径。

| 场景 | Prompt 长度 | Prefill（PP） | Decode（TG@128） |
|---|---:|---:|---:|
| 短上下文 | 2048 | 810.81 t/s | 55.29 t/s |
| 中上下文 | 8192 | 675.75 t/s | 55.06 t/s |
| 长上下文 | 16384 | 544.45 t/s | 55.56 t/s |
| 32k 极限测试 | 32768 | 472.37 t/s | 37.03 t/s |

可观察到：2048→16384 prompt 时，prefill 约下降 33%，decode 基本维持 55 t/s；32k 极限测试的 decode 降至约 37 t/s，不能代表日常单轮对话。

### 4.3 并发吞吐

| Parallel | Prefill | 单路 Decode | 聚合 Decode |
|---:|---:|---:|---:|
| 1（bench） | 810.81 t/s | 55.29 t/s | 55.29 t/s |
| 1（batched-bench） | 1005 t/s | 55 t/s | 55 t/s |
| 2 | 1102 t/s | 35 t/s | 70 t/s |
| 3 | 1087 t/s | 28 t/s | 83 t/s |
| 4 | 1157 t/s | 23 t/s | 90 t/s |

单路 Agent 延迟优先使用 `-np 1`；并发 2–4 路时聚合吞吐上升，但单路 decode 明显下降。

### 4.4 Prompt cache 实测收益

| 场景 | Prompt eval | 相对首次请求 |
|---|---:|---:|
| 首次完整请求 | 2328.31 ms / 1725 tokens | 基线 |
| 同会话历史复用 | 253.44 ms / 33 tokens | 约 9.2×，节省约 89.1% |
| 跨会话复用相同 system prompt | 201.24 ms / 16 tokens | 约 11.6×，节省约 91.4% |

## 5. 从硬件到模型的部署全链路

该流程只把原始记录能够支撑的步骤写成可复现检查项；没有记录的版本、下载地址和命令不作虚构。

### N0：确认硬件与可用内存边界

1. 记录设备型号、CPU/APU、核心线程数、Radeon 8060S、物理内存容量和内存速率。
2. 记录 Vulkan 驱动版本和所有 heap；不要把 32GB、15.57GB、64GB 直接相加后当作连续显存。
3. 用 ROCm/HIP 后端启动最小模型，确认日志识别 `gfx1151`，并记录实际可见 VRAM。
4. 先按“模型文件 + 满上下文 KV + 计算图 + batch/ubatch + 系统/驱动”估算预算，再决定量化和是否全量 offload。

### N1：确认 OS、固件与 AMD 驱动

1. 在目标机记录 Windows 版本/build、BIOS 版本及电源/内存配置；这些字段在原始记录中缺失，正式复测必须补齐。
2. 安装或确认 AMD 显示驱动 `32.0.12079.1014`（2025-08-01）及 Vulkan 1.3.292，重启后再次记录版本。
3. 若使用 ROCm/HIP，确认所用 ROCm llama.cpp 包确实包含 `gfx1151` 支持；ROCm 完整版本、安装来源和环境变量需要在下一次复测中锁定。

### N2：准备推理框架与后端

1. 准备并记录不可变版本或 hash：llama.cpp b9754 Vulkan、AMD ROCm llama.cpp（原始失败记录标识 b8407）以及 Step-3.7 使用的 stepfun Vulkan fork。
2. 按模型选择后端：gpt-oss 长上下文走 Vulkan；Qwen3.6、Qwen-AgentWorld、Qwen3-Next、GLM 走 ROCm/HIP；Step-3.7 Q3_K_L 走 stepfun Vulkan。
3. 不要因为模型能被加载就判定兼容；必须继续做 prompt eval、decode、实际 chat/server 和内存检查。

### N3：准备模型与分片

1. 下载与量化版本匹配的 GGUF 及全部分片，保存模型 revision、文件清单和 hash。
2. 多分片模型从 `00001-of-...` 加载；缺少第一片或从中间分片启动会直接失败。
3. 对照第 3 节的模型内存、KV 和上下文预算选择量化；Step-3.7 Q4_K_S 已实测全量 offload OOM，不应作为默认方案。

### N4：启动本地推理服务

1. 按模型使用对应后端和 offload 层数：gpt-oss `-ngl 99`，Qwen3.6/AgentWorld `41/41`，Qwen3-Next `49/49`，GLM `48/48`，Step Q3_K_L `-ngl all --cache-ram 0`。
2. gpt-oss 按第 4.1 节命令启动；生产单用户使用 `-np 1`，多人测试要重新测量单路延迟和内存。
3. 服务绑定地址、端口和路径使用部署环境变量或配置文件，不把本机 IP、用户目录和模型权重提交到仓库。

### N5：验收推理结果

依次完成以下检查，任一项失败都不能只用“纯 decode 能跑”作为通过依据：

```text
1. 进程启动并稳定监听
2. /health 返回正常
3. /v1/chat/completions 完成最小文本请求
4. llama-bench 同时测 PP 和 TG
5. 真实 prompt 经过 server/chat 路径，不出现退出或乱码
6. 记录模型、KV、计算图、系统和驱动的内存峰值
7. 用固定 Agent smoke test 检查工具调用、结构化输出和长上下文行为
```

原始记录已验证 `/health` 和 OpenAI-compatible API 形态，并完成 gpt-oss 的 Windows 64 位上层 Agent 客户端接入；但固定质量集、重复次数、波动范围和工具调用成功率尚未建立。

### N6：上层 Agent 接入

1. 以 OpenAI-compatible `/v1/chat/completions` 作为上层入口。
2. gpt-oss 的 reasoning level 写入 system message：`Reasoning: low/medium/high`；日常 Agent 原始建议为 `medium`，低延迟问答用 `low`，复杂规划/代码任务用 `high`。
3. 接入后重新验证首 token 延迟、decode、上下文复用和工具调用，不能直接沿用 `llama-bench` 的纯后端结果。

## 6. 质量与兼容性结论

### 质量证据边界

- 当前“很强 / 强 / 中高 / 综合最佳”是原始部署记录中的定性判断，或结合模型定位的能力描述。
- 已有本机端到端接入证据的是 `gpt-oss-120b Q4_K_M`（Windows 64 位上层 Agent 客户端）。其他模型主要有后端可运行和速度证据，尚未有同一提示词集上的质量对比。
- 下一轮若需要发布“质量排名”，必须补充固定 prompt、参考答案、工具调用成功率、结构化输出通过率和长上下文检索正确率。

### 选择建议

| 使用目标 | 首选 | 原因 |
|---|---|---|
| 综合 Agent、长上下文和本机稳定性 | gpt-oss-120b Q4_K_M + Vulkan | 128k、单路约 55 t/s 级 decode，工具调用生态和本机接入证据最完整 |
| 速度/体积平衡 | Qwen-AgentWorld-35B-A3B Q5_K_M + ROCm/HIP | 约 24.7GB，PP 最高约 1177 t/s，TG 约 48/45/42 t/s |
| 较低内存的工具/代码 Agent | Qwen3.6-35B-A3B Q4_K_M + ROCm/HIP | 约 20.6GiB，TG 约 50 t/s，Vulkan 路径暂不稳定 |
| 更大模型能力候选 | Qwen3-Next-80B-A3B Q6_K + ROCm/HIP | 约 62.2GB，能全量 offload，但速度低于 35B 系列 |
