# ComfyUI 模型部署通用节点式 SOP（新手执行版）

> 一句话定位：本 SOP 教第一次部署者判断目标模型能否由 ComfyUI 接入，并把它部署成可通过 UI 或 API 执行、可复现、可验收、可回滚的推理 workflow。

## 0. 新手入口：这份文档是做什么的

### 0.1 文档用途与适用对象

本 SOP 是一份面向**第一次使用 ComfyUI 部署模型的执行者（包括 AI 执行代理）**的操作说明。它指导执行者从“拿到一个目标模型和一台信息尚未确认的设备”开始，依次完成：

```text
确认模型要做什么
→ 判断模型如何接入 ComfyUI
→ 查询真实设备和软件平台
→ 证明整套版本兼容
→ 下载并安装 ComfyUI
→ 接入节点、模型和 workflow
→ 实际推理、验收、发布和回滚
```

读完并执行本 SOP 后，执行者应能回答三个问题：

1. 目标模型是否可以通过 ComfyUI 部署；如果不能，阻断发生在哪一层？
2. 当前设备可以接受哪一组硬件后端、驱动、Python、推理框架、ComfyUI、节点和模型版本？
3. 如何用已锁定的环境和 workflow 完成可复现、可验证、可回滚的推理？

本 SOP 是通用流程，不是硬件或版本答案库。正文不预填设备、驱动、框架、ComfyUI、节点或模型版本；实际部署时必须查询真实值，并写入本次 `deployment_record`。禁止猜测缺失值，也禁止照抄其他设备的版本组合。

### 0.2 ComfyUI 是什么，为什么可以用它部署模型

ComfyUI 是一个用节点图（workflow）组织和执行生成式模型推理的工具。执行者在 workflow 中连接模型加载、条件编码、采样/推理、解码、后处理和保存等节点；ComfyUI 按图调度这些节点，并通过底层推理框架和硬件后端完成计算。workflow 既可在可视化界面中运行，也可转换为 API workflow 后由程序提交。

ComfyUI 位于模型文件和底层计算环境之间，它不是显卡驱动、硬件运行时或推理框架：

```text
用户或调用程序
→ ComfyUI UI / API
→ workflow（原生节点和/或 custom node）
→ loader 读取模型及配套组件
→ Python / 推理框架 / 加速运行时
→ OS / 活动驱动 / 目标硬件
→ 推理输出、日志和指标
```

因此，只有当 ComfyUI 存在能加载并执行该模型的节点路径，并且底层设备、驱动、框架、精度和算子相互兼容时，才可以使用 ComfyUI 部署该模型。ComfyUI 能启动不代表目标模型一定受支持。

### 0.3 本文所说的“模型部署”

本文默认处理**推理部署**。部署是指把指定版本的模型文件及其组件、节点、workflow 和参数，绑定到一套已验证的设备、驱动、推理框架和 ComfyUI 环境中；随后通过 ComfyUI UI 或 API 实际完成推理，证明输出、资源、性能、稳定性和回滚满足预先定义的标准。

本 SOP 的硬件兼容主路径默认模型计算发生在目标设备。若 ComfyUI 节点只调用厂商托管 API，必须把任务标记为“远程服务接入”：目标设备只承载 ComfyUI/客户端，不能把云端模型能力归因于本机硬件兼容；还必须另验 API 鉴权、网络、数据出境与隐私、计费、限流、服务可用性和供应商回退。远程服务接入成功不能表述为“模型已在本机部署”。

以下情况都不能单独算作“部署完成”：

- 只下载或复制了模型文件；
- 只安装并打开了 ComfyUI 页面；
- 模型出现在下拉列表中，但 workflow 未成功执行；
- 生成过一次输出，但实际计算设备或后端不明确；
- 使用 `latest`、默认分支或未锁定依赖，结果无法复现；
- 尚未验证重启、重复运行、稳定性和回滚。

如果目标包含训练、微调、模型转换、量化或分布式执行，必须在 N0 明确记录；不能把这些工作默认当作普通 ComfyUI 推理部署。

### 0.4 模型接入 ComfyUI 的两条路径

在 N0 必须先初步判断接入路径，N1 再结合候选 ComfyUI 版本确认兼容范围。不能把任意权重文件直接丢入模型目录后期待 ComfyUI 自动识别。

| 接入路径 | 何时使用 | 要完成的动作 |
|---|---|---|
| ComfyUI 原生支持 | ComfyUI 内置节点已能加载模型格式并表达官方推理流程 | 按模型类别放置组件，使用内置 loader 和节点导入或构建 workflow |
| custom node / wrapper 支持 | 模型需要专用 loader、预后处理、算子或流程 | 审查并锁定必要 custom node；若无现成节点，只能走获批的 wrapper 开发与独立验证 |

如果两条路径都不存在，或路径所需算子无法在目标后端运行，当前方案应标记为 `BLOCKED`，返回 N0/N1 调整模型、后端或交付方式，不能伪装成已部署。

### 0.5 开始时需要什么，完成后交付什么

开始执行时，至少要有可追查的目标模型或用途、可查询的目标设备，以及执行查询、下载、安装、启动和保存证据的权限。模型 revision、workflow、硬件详细信息和版本组合可以在 N0/N1 查明；凡是会改变部署路线的未知项，在查明前必须阻断后续操作。

| 输入或交付物 | 最低要求 |
|---|---|
| 初始输入：目标模型与任务 | 知道模型来源或待解决的任务，并能说明希望输入什么、输出什么 |
| 初始输入：目标设备与边界 | 可查询真实主机；明确测试/生产、UI/API、联网/离线和权限边界 |
| 交付：可运行实例 | 使用已批准环境，从停止状态启动 ComfyUI，并通过 UI 或 API 完成代表性 workflow |
| 交付：唯一绑定 | 设备、后端、运行时、ComfyUI、节点、模型、workflow 和参数均能追溯到 `deployment_id` |
| 交付：验收与恢复 | 有功能、质量、资源、性能、稳定性、停止、重启和回滚结果 |
| 交付：证据 | 有版本清单、来源、hash、原始日志、输出、问题记录和最终结论 |

最终交付的不是“一个能打开的 ComfyUI 目录”，而是一套**可说明、可运行、可复现、可验证、可回滚**的模型部署绑定。

### 0.6 第一次部署的最短路径

六个主节点就是完整执行顺序，不需要把下面每一行再拆成新的流程节点：

| 顺序 | 先做什么 | 本节点完成的标志 |
|---|---|---|
| N0 需求定义 | 阅读模型官方说明和示例，列出组件、输入输出、节点/算子和验收标准；判断原生或 custom node 接入路径 | 知道“用什么模型，在什么输入下得到什么输出，怎样算成功” |
| N1 设备、平台与兼容性 | 查询真实硬件、OS 和活动驱动；选择主后端；用官方资料求出整套兼容版本交集 | 得到有证据的可接受版本集合、首选元组和回滚元组 |
| N2 资源、版本、下载与安装 | 检查容量、网络和权限；锁定 ComfyUI 版本/制品；校验后在隔离环境安装 | 获得身份可复核、依赖干净的 ComfyUI 基础实例 |
| N3 运行时与算子能力 | 确认框架在批准设备上计算；实测模型所需精度、算子和内存；启动基础 ComfyUI | 设备落点正确，无未批准 fallback，基础运行时通过 |
| N4 节点、模型与 workflow 接入 | 安装必要节点，校验并放置全部模型组件，导入/创建 workflow，执行固定输入 | ComfyUI 通过 UI 或 API 生成有效且可追溯的端到端输出 |
| N5 综合验收、发布与回滚 | 做冷启动、重复/并发/稳定性、停止重启和回滚，冻结全部清单 | 达到预设指标，并形成可复现、可恢复的批准或拒绝结论 |

### 0.7 新手术语速查

| 术语 | 在本 SOP 中的含义 |
|---|---|
| checkpoint / diffusion model | 执行主要生成或变换计算的模型权重；不一定包含所有配套组件 |
| text encoder / tokenizer | 把文字切分并编码为模型可用条件的组件 |
| VAE / decoder | 在潜空间表示与可查看图像等输出之间转换的组件 |
| CLIP Vision / image encoder | 把参考图像编码为条件特征的组件 |
| LoRA / ControlNet 等附加模型 | 修改或约束主模型行为的附加权重，不能代替主模型 |
| loader | 读取特定模型类别、格式和组件的节点；loader 不匹配时，文件存在也无法正确加载 |
| sampler / scheduler | 控制扩散或迭代推理过程及步进策略的节点/参数 |
| workflow | ComfyUI 的节点、连接和参数图；它决定模型实际如何执行 |
| UI workflow JSON | 用于在界面中继续编辑的图结构，通常包含布局信息 |
| API workflow JSON | 用于 API 提交的可执行结构，不能假设与 UI JSON 可直接互换 |
| custom node | 安装到 ComfyUI 的第三方或内部扩展代码，可能带来额外依赖和后端限制 |
| wrapper | 把尚未原生接入的模型调用封装成 ComfyUI 节点/接口的适配代码；它也必须锁版本并单独验证 |
| revision / commit / digest | 用来唯一标识模型、代码或镜像内容的不可变身份；分支名和 `latest` 不属于可靠锁定 |
| dtype / 量化 | 模型计算或存储所用的数值精度/压缩表示，会直接影响兼容性、内存、速度和质量 |
| backend / runtime | 把框架计算提交到目标硬件的后端及运行时；必须与硬件、驱动和框架构建匹配 |
| fallback | 原路径不可用时转到其他设备或实现；未获批准的静默 CPU fallback 视为失败 |
| smoke | 用最小代表性输入快速证明服务、模型、workflow 和输出链路可工作的测试 |
| hash / manifest | hash 用于核对文件内容；manifest 汇总本次部署所有绑定对象、版本、配置和证据 |
| deployment_record_id | 从 N0 到 N5 一次部署活动的记录标识，可包含多个被比较或淘汰的候选方案 |
| candidate_id | N0/N1 中一个候选接入路线和版本元组的标识；被淘汰时保留记录，不等同于部署批准 |
| deployment_id | N1 选定最终候选、进入 N2 前分配的不可变部署身份；之后绑定字段变化必须按规则生成新 ID |
| SLO / 验收阈值 | 测试前约定的成功率、质量、耗时、吞吐、资源、稳定性或恢复目标 |
| 模型/输入/输出目录 | 分别保存被 loader 扫描的模型组件、任务输入和生成结果；模型必须放入 loader 对应类别或受控映射路径 |

### 0.8 如何执行本文

1. 创建唯一 `deployment_record_id` 和第 9 节建议的证据目录；N0/N1 的候选方案分别使用 `candidate_id`，此时不要提前分配 `deployment_id`。
2. 严格按 N0 → N5 执行，不要从下载模型或安装节点直接开始。
3. 每个节点依次完成“实际执行步骤 → 查询和填写 → 强制校验 → 保存证据 → 判定状态”。
4. 当前节点只有达到 `PASS`、有效 `CONDITIONAL_PASS` 或获批 `N/A`，才可进入下一节点。
5. 出现问题时使用本节点的问题建议；如果处理动作改变任何绑定字段，按第 6 节返回受影响节点重验。
6. N0-N5 完成后执行第 7 节最终放行检查。不得把单次成功输出写成最终批准。

AI 执行代理可以查询、比较、安装、测试和留证，但不能自行发明兼容结论、验收阈值、许可证授权或风险批准。缺少会改变路线的决定时，应列出缺项和影响，将节点标记为 `BLOCKED`，交由本次记录中的 Owner/审批人确认。

## 1. 执行规则

### 1.1 强绑定原则

每次部署必须绑定以下完整元组：

```text
部署需求
→ 硬件/固件/内存
→ OS/驱动/推理后端
→ Python/推理框架/加速运行时
→ ComfyUI/节点/依赖
→ 模型/权重/dtype/量化
→ workflow/参数
→ smoke/性能/稳定性/回滚
```

任一绑定字段变化，必须生成新的变更记录，并按第 6 节重新验收。

N0/N1 尚未安装任何候选栈时，可以在同一 `deployment_record_id` 下用不同 `candidate_id` 比较并淘汰方案。N1 批准最终版本元组、准备进入 N2 时才分配 `deployment_id`，并冻结计划安装的绑定身份；此后任何已冻结绑定字段变化都必须生成新的 `deployment_id`。N5 冻结的是同一 `deployment_id` 实际通过验收的制品、配置和证据，不是再次分配 ID。仅补充证据而实际对象未变化时，可保留原 ID 并记录 `change_id`。

### 1.2 节点状态

| 状态 | 含义 | 能否进入下一节点 |
|---|---|---|
| `NOT_STARTED` | 尚未执行 | 否 |
| `IN_PROGRESS` | 查询或证据未完成 | 否 |
| `PASS` | 全部强制校验通过 | 是 |
| `CONDITIONAL_PASS` | 实验性路径或存在已审批偏离 | 仅在批准范围和有效期内允许 |
| `BLOCKED` | 不支持、冲突、证据缺失或验证失败 | 否 |
| `N/A` | 经复核后确实不适用 | 是，但必须填写理由 |

未知结果不能写成 `N/A`。当前节点未达到 `PASS`、有效 `CONDITIONAL_PASS` 或获批 `N/A` 时，不得进入下一节点。

节点记录中，单项校验可以写 `PASS`、`FAIL` 或经确认的 `N/A`；只要存在一个未解决的 `FAIL`，该节点总状态就是 `BLOCKED`，不能把单项失败直接当作新的节点状态。

### 1.3 证据要求

| 证据 | 用途 |
|---|---|
| 官方支持矩阵、安装说明和 release notes | 证明官方支持边界 |
| tag、完整 commit、digest、签名和发布元数据 | 证明对象身份 |
| 本机原始查询输出 | 证明实际环境 |
| 算子和设备测试 | 证明底层运行能力 |
| 固定模型和 workflow smoke | 证明模型绑定可运行 |
| 稳定性、生产回归和回滚记录 | 证明可交付 |

命令本身不是证据；带时间戳的原始输出、日志、快照和摘要才是证据。

## 2. 六个主节点

```text
N0 需求定义
→ N1 设备、平台与兼容性
→ N2 资源、版本、下载与安装
→ N3 运行时与算子能力
→ N4 节点、模型与 workflow 接入
→ N5 综合验收、发布与回滚
```

| 节点 | 汇总内容 | 主要输出 |
|---|---|---|
| N0 | 业务目标、模型身份、负载、SLO、许可证和风险 | `deployment_request.yaml` |
| N1 | 硬件、OS、驱动、后端路由、官方兼容矩阵、批准版本集合 | `compatibility_profile.yaml` |
| N2 | 容量、网络、发行形态、版本锁、下载完整性、隔离安装 | `environment_manifest.yaml` |
| N3 | 解释器、框架、设备落点、dtype、算子、内存能力和基础健康 | `runtime_validation.yaml` |
| N4 | custom node、模型资产、依赖、workflow、参数和端到端输出 | `model_binding.yaml` |
| N5 | smoke、性能、稳定性、停止、恢复、生产晋级和回滚 | `release_acceptance.yaml` |

## 3. 硬件后端分流参考

硬件分流在 N1 内完成，不单独形成更多流程节点。

| 硬件类型 | 候选后端 | 必须确认 |
|---|---|---|
| NVIDIA GPU | PyTorch CUDA | 计算能力、活动驱动、CUDA 构建、目标 kernel |
| AMD GPU/APU | PyTorch ROCm/HIP | 精确产品/架构、OS 分支、驱动、ROCm/PyTorch 矩阵和限制 |
| Intel GPU | PyTorch XPU 或厂商正式路径 | GPU 代际、驱动、XPU 构建和算子覆盖 |
| Apple Silicon | PyTorch MPS | macOS、芯片、统一内存、PyTorch 和目标算子 |
| CPU | PyTorch CPU | CPU 架构、指令集、wheel、内存和性能 |
| NPU/MLU | 厂商框架扩展、执行提供程序或已验收节点 | SDK、驱动、模型格式、图编译、量化和 fallback |
| 其他加速器 | 厂商正式支持的框架和 ComfyUI 接入 | 无正式或内部验收接入时不得自动映射 |

DirectML、社区后端、转换层和社区容器必须作为独立候选记录，不能等同于厂商正式后端。

## 5. 主节点校验

### N0：需求定义

#### 目标

在查询设备和选择版本前，明确模型身份、工作负载、性能、质量、许可证和允许的风险。

#### 实际执行步骤

1. 先明确目标是“模型在目标设备本地推理”还是“ComfyUI 调用远程托管 API”，并记录计算发生位置、数据流向和交付边界。
2. 检查官方仓库的实际文件树和大文件类型，确认其中确有模型权重/组件、推理实现、依赖、配置和许可证；克隆成功、仓库很大或存在 LFS 文件都不能替代这项检查。
3. 获取模型官方仓库、模型卡、许可证、示例 workflow 或官方推理代码，并记录不可变 revision。
4. 根据官方说明列出主模型、编码器、tokenizer、VAE、ControlNet/LoRA 等必需组件及输入输出要求；不适用的组件写明理由。
5. 打开示例 workflow 或推理代码，整理所需 loader、节点、算子、外部服务和特殊 kernel。
6. 从模型方 workflow/推理代码提取所需 loader、class type 和流程能力，再对照 ComfyUI 官方核心节点说明，形成“原生支持候选”或“custom node 候选”；此处不预设最终 ComfyUI 版本，版本级确认在 N1 完成。两类候选都不存在时，记录 wrapper 开发需求并暂停现有部署路线。
7. 定义目标规格和验收标准，包括尺寸/时长、batch、并发、质量、耗时、吞吐、稳定性和允许降级范围。
8. 填写 N0 记录；仍有会影响设备、后端或版本选择的未知项时，将状态标记为 `BLOCKED`，不得进入 N1。

#### 需要查询和填写

- 模型仓库、不可变 revision、文件格式、dtype 和量化格式。
- 仓库是本地模型制品、推理代码、文档/演示仓库还是托管 API 接入仓库。
- 本地推理或远程服务边界、实际计算位置、数据流向、鉴权、计费和可用性要求。
- 任务类型、输入输出和 workflow。
- 最低、目标和最大尺寸、帧数、时长、batch 和并发。
- 成功率、耗时、吞吐、质量、稳定性和恢复目标。
- custom node、外部服务、特殊 kernel 和后端限制。
- 是否需要训练、分布式、混合卸载或 CPU fallback。
- 许可证、鉴权、网络和数据安全要求。
- 是否接受实验性路径、风险 Owner 和回滚目标。

#### 必须校验

- 模型和依赖锁定到不可变 revision，禁止只写简称、`latest` 或默认分支。
- 仓库文件清单中存在与目标任务匹配的真实模型组件和推理实现；教程视频、样例输出、网页、LFS 指针或通用 LFS 规则不能冒充模型权重。
- “本地模型部署”和“远程 API 接入”没有混淆；远程 wrapper 只能证明服务接入，不能证明模型与本机推理栈兼容。
- 所需组件、节点、算子和外部依赖全部列出。
- 原生或 custom node 候选路径有可复核依据；只有相似节点名或模型文件扩展名相同，不能作为支持证据，最终版本级兼容结论由 N1 给出。
- 输入范围和验收阈值可以测量。
- 许可证和使用范围已批准。
- 训练或分布式需求没有被误当作普通 ComfyUI 推理。

#### 完成条件与证据

- 必填项完整，不存在会改变后端选择的未知项。
- 已定义成功、失败和允许降级标准。
- 证据：部署申请、模型卡、revision、许可证、workflow 和能力清单。

#### 常见问题与建议

| 问题现象 | 可能原因 | 解决建议 | 重新验收条件 |
|---|---|---|---|
| 只提供模型名称 | 模型身份未锁定 | 获取仓库、revision、组件和许可证 | 模型身份与依赖完整 |
| 仓库已克隆但没有权重或推理代码 | 克隆的是产品介绍、文档、演示或 API 仓库 | 核对文件树和模型卡；向官方取得本地制品，或把任务改为远程服务接入 | 权重、组件、推理实现和许可证齐全，或新的 API 接入契约获批 |
| 最大文件是 MP4/样例输出/HTML | 大文件被误当模型制品 | 校验文件类型、LFS 对象、大小和 hash；禁止放入模型目录 | 每个模型组件与官方清单精确匹配 |
| 只有云端 API，没有本地权重 | 模型未开放本地部署 | 停止本地兼容选型；如接受云服务，另验鉴权、数据、计费、限流、可用性和回退 | 远程服务接入范围获批，且不再表述为本地模型部署 |
| 无法列出节点或算子 | workflow 缺失或来源不明 | 解析真实 workflow 和官方推理代码 | 依赖与算子清单完成 |
| 模型支持 GPU，但节点只支持某厂商 | 模型和节点要求混淆 | 单独审查节点 wheel、扩展和编译依赖 | 返回 N0/N1 重新选路 |
| 只有“能运行”，没有规格 | 缺少可测目标 | 补充负载、性能、质量和恢复指标 | 契约获业务和技术复核 |
| 下载需要登录 | 授权未准备 | 完成许可证、凭据托管和日志脱敏 | 授权及访问测试通过 |

### N1：设备、平台与兼容性

#### 目标

一次完成硬件识别、OS/驱动确认、后端选择和全栈兼容版本交集。

#### 实际执行步骤

1. 使用操作系统和厂商工具采集 CPU、内存及全部加速器的型号、设备 ID、架构、显存/共享内存和拓扑。
2. 采集 OS、架构、内核/构建号、活动驱动、固件及设备健康状态，并区分“已安装版本”和“当前实际加载版本”。
3. 根据 N0 的模型、节点和算子要求列出候选推理后端，选定一个主后端；备用或 fallback 路径单独记录，不与主后端混装。
4. 查询硬件厂商、推理框架、ComfyUI、custom node 和模型维护方的官方兼容资料，保存 URL、访问时间、相关原文和页面快照摘要。
5. 把 OS、驱动、runtime、Python、框架、ComfyUI、节点和模型格式组成候选版本元组，逐层求共同支持的交集，而不是分别判断各组件“看起来兼容”。
6. 从交集中确定 `acceptable_version_set` 和首选元组，并标明支持等级；已有批准部署时同时确定回滚元组，首次部署则记录“部署前状态或 N2 干净基线”作为回滚目标。交集为空、证据冲突或设备身份不清时停止并重新选路。

版本交集按下面的关系计算并逐行留证：

```text
N0 模型/workflow 要求
∩ custom node 或原生节点支持范围
∩ ComfyUI core/frontend 支持范围
∩ Python/推理框架/加速 runtime 支持范围
∩ OS/活动驱动/固件/精确硬件支持范围
= acceptable_version_set
```

任一层没有公开矩阵时，必须记录“缺少哪项官方交叉关系”，再用本机专项验证和审批补证；不能把未知范围直接当作全集。

#### 需要查询和填写

- 主机、主板、BIOS/固件、CPU 和虚拟化方式。
- 枚举全部 GPU/NPU/加速器、物理 ID、设备 ID、架构和多卡拓扑。
- 物理内存、OS 可见内存、独显、共享内存、UMA、pagefile/swap。
- OS、构建号/内核、架构和补丁状态。
- 活动驱动、驱动包、模块/INF、固件、签名和设备健康。
- 候选 CUDA、ROCm、XPU、MPS、CPU、NPU/MLU 等后端。
- 厂商、框架、ComfyUI、模型和节点的官方支持矩阵。
- 候选版本集合、首选元组、支持等级，以及适用时的回滚元组；首次部署填写非版本式回滚目标。

#### 必须校验

- 所有物理加速器均被识别，虚拟显示设备没有被误判。
- 硬件身份达到官方矩阵要求的粒度。
- OS、活动驱动、固件和设备可建立官方或已审批关系。
- 只选择一个主后端；备用后端单独记录。
- 不混装不同后端的框架构建。
- 模型必要算子与候选后端不存在已知硬冲突。
- N0 的接入候选已与每个候选 ComfyUI 版本的实际核心节点范围或固定 custom node 版本逐项核对。
- 完整版本元组必须求交集，不能根据相似型号或“版本更高”推断。
- 正式部署禁止浮动分支、`latest` 和未锁定 nightly。
- 官方、实验性、社区和内部支持分级记录。

#### 完成条件与证据

- 设备、平台和后端可唯一识别。
- 形成 `acceptable_version_set` 和首选元组；已有批准部署时形成回滚元组，首次部署时形成明确的非版本式回滚目标。
- 无未解释的 Unknown；偏离有 Owner、范围、补偿测试和到期日。
- 证据：硬件/平台清单、活动驱动、官方矩阵快照、版本交集和路由审批。

#### 常见问题与建议

| 问题现象 | 可能原因 | 解决建议 | 重新验收条件 |
|---|---|---|---|
| 只显示通用 GPU 名称 | 驱动缺失或采集粒度不足 | 用设备 ID、PCI ID、固件和厂商工具交叉识别 | 精确命中产品/架构 |
| 多 GPU 不知道使用哪张 | 显示和计算设备混淆 | 绑定物理 ID、逻辑编号和主/备设备 | 设备映射可复核 |
| 各工具内存数值不同 | 独显、UMA、地址空间口径不同 | 分口径和采样时间记录，禁止直接相加 | 内存拓扑解释完成 |
| 厂商软件和活动驱动版本不同 | 安装包不等于加载版本 | 分别记录，以活动模块/INF 为准 | 重启后重新采集 |
| OS/硬件不在矩阵 | 版本过新、过旧或尚未支持 | 更换受支持组合或走条件审批 | exact row 或完整例外证据 |
| 组件单独支持但组合无版本 | 完整版本交集为空 | 调整驱动、后端、框架、ComfyUI、节点或模型格式 | 新元组全部有依据 |
| NPU 可见但 ComfyUI 不可用 | 无适配节点、格式或算子覆盖 | 使用正式接入，否则回退其他后端 | 目标模型最小图通过 |
| 非 NVIDIA API 名称显示 `cuda` | 框架复用兼容命名 | 以构建标识、runtime 和矩阵判定后端 | 后端身份多源一致 |

### N2：资源、版本、下载与安装

#### 目标

一次完成资源预检、ComfyUI 发行版本锁定、下载完整性和隔离安装。

#### 实际执行步骤

1. 按下载、解压、依赖缓存、模型、临时文件、输出和回滚副本计算完整空间预算，并检查目录权限、文件系统能力和内存余量。
2. 验证批准下载源、最终 CDN、代理、TLS、认证和离线转运路径，确认能完整取得所需制品且凭据不会进入日志。
3. 按 N1 批准元组选择 ComfyUI 发行形态，把版本锁定到 tag 对应的完整 commit、安装资产或容器 digest，并关闭未受控自动更新。
4. 从批准来源下载 ComfyUI 及基础制品；校验资产名、字节数、可信 hash、签名和归档完整性，生成下载清单后才允许解压/安装。
5. 创建独立且可识别版本的安装目录和 Python 环境，在该环境中安装锁定依赖；不得复用未经验证的旧环境或系统全局包。
6. 执行依赖一致性检查，保存解释器绝对路径、包清单和安装日志，并确认基础实例中没有未批准模型、custom node、源码修改或残留文件。

发行形态只按本次兼容证据和维护方式选择：

| 发行形态 | 可以选择的前提 |
|---|---|
| Desktop / Portable | 目标 OS/硬件受支持，内置运行栈落在 N1 的版本交集内，并且能锁定版本和更新行为 |
| source + 独立环境 | 需要精确控制 Python/框架/依赖，且执行者能锁定 commit、依赖和启动配置 |
| container | 目标后端正式支持容器及设备透传，并能锁定镜像 digest、宿主驱动和挂载/权限 |
| 内部发行 | 来源、构建过程、补丁、依赖和摘要均可审计，且支持范围已有批准 |

发行形态没有天然优先级；不能因为安装方便就绕过 N1 的版本交集。

#### 需要查询和填写

- 下载、解压、安装、依赖缓存、模型、输出、临时空间和回滚预算。
- 系统/设备内存、UMA、pagefile/swap 和批准余量。
- 安装、模型、缓存、日志和输出目录及权限。
- DNS、TLS、代理、认证、重定向 CDN 和离线转运方案。
- Desktop、Portable、source、container 或内部发行形态。
- version、tag、完整 commit/镜像 digest、frontend 和更新行为。
- 固定 URL、资产名、字节数、发布者 hash、签名和 release 元数据。
- Python 隔离方式、依赖锁、软件源、端口和运行账户。

#### 必须校验

```text
所需空间 =
下载资产 + 解压/安装 + 依赖缓存 + 模型
+ 临时空间 + 回滚副本 + 输出保留 + 批准安全余量
```

- 容量、目录读写、文件系统和权限满足完整预算。
- 所需来源和最终 CDN 可达，TLS 校验未关闭，凭据不进入日志。
- 发行形态与 N1 批准元组一致。
- tag 解析到完整 commit，容器锁定 digest。
- 禁止浮动 URL、默认分支和非受控自动更新。
- 下载结果的字节数、可信摘要、签名和归档完整性通过。
- 发布方无摘要时按组织首次入库流程处理，不能把本地 hash 冒充发布者证明。
- 新实例不复用未经验证的旧环境或全局包。
- Python/包管理器属于当前实例，依赖一致性检查通过。
- 基线无未批准节点、模型、源码修改或残留文件。

#### 完成条件与证据

- 资源、网络、安全、版本身份、下载完整性和干净安装全部通过。
- 生成 `version_lock`、`download_manifest`、依赖锁和基础实例清单。
- 证据：容量表、网络/TLS、release 元数据、hash/签名、安装日志、绝对路径和包清单。

#### 常见问题与建议

| 问题现象 | 可能原因 | 解决建议 | 重新验收条件 |
|---|---|---|---|
| 下载前有空间，解压失败 | 未计算临时/缓存/回滚 | 按完整公式重新预算 | 下载与安装卷均通过 |
| 主页可访问但资产失败 | CDN、代理或认证未处理 | 记录最终域名，按制度配置网络和凭据 | 完整重定向链通过 |
| TLS 失败 | 时间、证书链或企业代理 | 修复受信链，禁止跳过校验 | 来源和证书验证通过 |
| URL 含 `latest` | 使用浮动地址 | 解析到 release、commit/digest 和固定资产 | 版本锁完整 |
| 大小/hash 不一致 | 中断、缓存、资产变化或风险 | 隔离文件，从批准来源重新获取 | 身份与完整性全量通过 |
| 包装到系统 Python | 调错解释器/包管理器 | 删除污染实例，用目标解释器重建 | 路径和包清单一致 |
| 安装节点前基线已不干净 | 复用旧目录或自动 patch | 从批准介质重建；补丁进入内部 commit | commit、依赖和文件清单干净 |
| 文件被安全软件隔离 | 误报或来源不受信 | 先核来源/hash，再按安全流程处置 | 恢复后重新计算摘要 |

### N3：运行时与算子能力

#### 目标

确认解释器、框架、加速后端和物理设备绑定正确，并验证目标模型需要的 dtype、算子和内存能力。

本节点验证“基础运行时和可独立测试的模型能力”，不在目标实例中正式绑定模型或第三方节点。可使用最小测试脚本、官方自测或从 N2 基线复制的临时验证环境；只有随 custom node/模型才能触发的专用算子，必须列入 `joint_validation`，并在 N4 的真实 workflow 中执行后记录 `PASS` 或 `FAIL`。

`joint_validation` 是转交给 N4 的跨节点校验项，不是节点状态。只有每个转交项都写明对象、原因、N4 验证方法和通过条件时，N3 才能对自身范围判定 `PASS`；N4 完成前不得遗留未判定的 `joint_validation`。

#### 实际执行步骤

1. 使用 N2 记录的绝对解释器启动验证，记录 Python、框架完整构建、加速 runtime 和依赖来源，确认没有调用其他环境。
2. 按物理设备 ID 配置目标设备，在框架中枚举设备，并核对逻辑编号与真实硬件的对应关系。
3. 在目标设备执行张量创建、传输、计算、同步和释放；结合进程与设备监控确认计算没有静默回退到 CPU 或其他未批准后端。
4. 根据 N0 的真实 workflow 整理 dtype、量化、attention、卷积、自定义 kernel 和代表性 shape；对可独立验证项逐项运行最小测试，对只能端到端触发的项目登记 `joint_validation` 和 N4 验证方法。
5. 检查 NaN、Inf、全零和异常饱和，并与参考结果比较；重复测试，记录首次、预热后耗时、峰值内存和缓存行为。
6. 用固定启动命令运行干净的 ComfyUI 基线，验证 UI/API 或健康接口可访问且启动日志无阻断错误；失败时返回 N1 或 N2 修正。

#### 需要查询和填写

- Python 绝对路径和版本。
- 框架完整构建、安装来源和加速 runtime。
- 驱动、设备数量、物理 ID、逻辑编号和设备选择变量。
- ComfyUI 启动命令、工作目录、监听配置和健康接口。
- 模型需要的 dtype、量化、attention、卷积、自定义 kernel 和代表性 shape。
- 允许的数值容差、fallback、耗时和内存阈值。

#### 必须校验

- 当前进程使用 N2 记录的解释器和依赖环境。
- 框架构建、runtime、活动驱动和设备符合批准元组。
- 设备枚举及物理映射准确。
- 在目标设备完成张量创建、传输、计算、同步和释放。
- 不存在未批准的静默 CPU fallback。
- 算子测试覆盖模型真实需要且可独立验证的 dtype、kernel 和代表性 shape；仅能随节点/模型触发的项目已进入 `joint_validation`，没有被误写为已通过。
- 输出无 NaN、Inf、全零或异常饱和，并满足参考容差。
- 覆盖首次和重复执行，记录编译缓存、峰值内存和耗时。
- ComfyUI 基线启动并通过健康检查。

#### 完成条件与证据

- 运行时、框架、后端和物理设备形成唯一绑定。
- N3 范围内的必要算子和资源预算有本机实测证据；待 N4 联合验证的项目有明确对象、方法和通过条件。
- 证据：运行时清单、设备落点、算子矩阵、参考比较、资源记录和启动日志。

#### 常见问题与建议

| 问题现象 | 可能原因 | 解决建议 | 重新验收条件 |
|---|---|---|---|
| 框架只看到 CPU | CPU 构建、设备屏蔽或初始化失败 | 安装批准构建，修正设备可见配置 | N2/N3 全部重验 |
| 设备顺序不符 | 多卡枚举变化 | 按物理 ID 绑定，避免只用序号 | 映射和实际落点一致 |
| DLL/SO 缺失 | 驱动/runtime 不兼容或路径污染 | 调整批准栈并清理冲突路径 | 返回 N1/N2 |
| 算子/精度不支持 | 后端能力或构建缺失 | 选择已验证精度/实现或调整栈 | 完整算子矩阵通过 |
| NaN、黑图或异常输出 | 低精度溢出或 kernel 缺陷 | 与高精度参考比较，调整精度或实现 | 数值和后续 workflow 重验 |
| 大 shape OOM | 超预算或碎片化 | 使用批准的分块/卸载或调整目标规格 | 目标规格重新验收 |
| 重复执行崩溃 | 缓存、泄漏或异步错误 | 增加同步和连续测试，修复 kernel/缓存 | 重复算子和稳定性通过 |
| 性能异常但无报错 | CPU fallback 或非优化路径 | 检查落点、利用率和编译日志 | 落点、耗时和稳定性重验 |

### N4：节点、模型与 workflow 接入

#### 目标

把 custom node、模型资产、依赖、workflow、参数和输出绑定为一个可复现的端到端对象。

#### 实际执行步骤

1. 先在目标 ComfyUI 基线中枚举实际可用的核心节点/loader，并与 N0 证据核对；原生支持时使用内置 loader，非原生支持时审查并锁定必要 custom node，需要 wrapper 时先完成适配和独立验证。
2. 安装 custom node 前保存依赖快照，只安装 workflow 必需的节点；安装后比较依赖差异，若框架或基础栈发生未批准漂移，立即停止并恢复 N2 基线。
3. 从固定 revision 下载主模型及全部配套组件，逐个校验文件名、字节数和 hash，排除 HTML、LFS 指针和未完成文件。
4. 把组件放入 loader 对应的模型类别目录，或配置受控的额外模型路径；重启/刷新后确认每个名称只解析到唯一文件。
5. 导入或创建 UI workflow；需要程序调用时同时导出 API workflow，并检查每个 class type、模型引用、节点连接和输入映射。
6. 使用冻结的测试输入、prompt、seed 和 N3 已验证范围内的参数执行最小端到端任务，保存任务历史、日志、资源记录和有效输出。

#### 需要查询和填写

- 节点仓库、维护方、许可证、commit、依赖、安装脚本和后端限制。
- 节点内部 patch 和依赖安装前后差异。
- 模型仓库、revision、组件关系、文件名、字节数、hash、dtype、量化和许可证。
- 主模型、编码器、VAE、tokenizer、Control 模型等组件。
- 模型目录类别和唯一解析路径。
- UI/API workflow、文件 hash、class type 和节点/模型引用。
- 输入文件 hash、prompt、seed、采样、尺寸、batch、帧数和输出标准。
- 参数 schema、默认值、fallback 和资源阈值。

#### 必须校验

- 只安装 workflow 必需节点，锁定不可变 commit。
- 节点代码、安装脚本、许可证和安全风险已审查。
- 节点依赖未造成未解释的框架或基础依赖漂移。
- 节点导入且 class type 可枚举。
- N3 的全部 `joint_validation` 项已在实际节点和模型路径中通过；失败时返回 N3/N4，不得仅凭 workflow 其他部分可运行而放行。
- 模型锁定 revision、字节数和 hash，不是 HTML、LFS 指针或半截文件。
- 所有必需组件齐全，模型格式与 loader 匹配；可独立验证的 dtype/量化能力已在 N3 通过，只能随节点/模型触发的能力已作为 `joint_validation` 在本节点通过。
- 目录映射唯一，无同名资产歧义。
- UI/API workflow 用途明确且可解析。
- 每个节点、模型和输入均解析到唯一绑定对象。
- 参数处于 N3 验证范围，workflow 可完整执行并返回有效输出。
- 输出可关联 `deployment_id`。

#### 完成条件与证据

- 节点、模型、依赖、workflow、参数和输出形成唯一绑定。
- 端到端功能、质量和资源指标满足批准标准。
- 证据：commit、依赖差异、模型 manifest、workflow/hash、任务历史、日志和输出。

#### 常见问题与建议

| 问题现象 | 可能原因 | 解决建议 | 重新验收条件 |
|---|---|---|---|
| 节点缺失/导入失败 | commit 不兼容、依赖缺失或装错环境 | 使用固定兼容节点和目标解释器 | N2/N4 重验 |
| 安装节点后框架变化 | requirements 过宽 | 恢复基础栈，为节点建立依赖锁或补丁 | N2-N4 重验 |
| 节点 API 不兼容 | ComfyUI 和节点单边升级 | 选择已验证组合或受控迁移 | 节点和 workflow 重验 |
| 模型文件很小/不可解析 | 登录页、LFS 指针或错误响应 | 从固定 revision 重新下载并校验 | 文件身份和完整性通过 |
| 模型未出现在列表 | 目录类别或扫描问题 | 修正唯一目录映射并受控重启 | 枚举和引用通过 |
| 同名加载错模型 | 多目录歧义 | 使用唯一路径和 hash 绑定 | 模型绑定和输出基线重验 |
| UI 可开但 API 不执行 | workflow 格式或输入映射错误 | 导出规范 API workflow 并锁 hash | 解析、提交和执行通过 |
| workflow OOM | 规格超 N3 验证范围 | 使用批准策略并返回 N3 验证 | 目标规格端到端通过 |
| 质量异常 | 精度、组件、参数或版本漂移 | 比较冻结 manifest，恢复批准绑定 | 全部标准用例重验 |
| 固定 seed 有差异 | 后端非确定性 | 量化差异并按批准容差判断 | 重复性和质量通过 |

### N5：综合验收、发布与回滚

#### 目标

验证部署可冷启动、持续运行、停止、恢复、监控、生产晋级和回滚。

#### 实际执行步骤

1. 在测试前冻结 smoke、稳定性和性能用例，以及成功率、质量、延迟、吞吐、资源和恢复阈值。
2. 从服务停止状态依次执行冷启动、健康检查、设备/节点/模型枚举、代表性 workflow、受控停止、再次启动和再次推理。
3. 按批准的持续时间、次数、规格和并发运行稳定性及性能测试，分别记录首次、预热、连续和并发结果。
4. 同时监控设备利用率、内存、进程、线程、队列、缓存、日志、临时文件和磁盘增长，确认无泄漏、死锁、设备重置或未批准 fallback。
5. 全部通过后冻结 deployment manifest、依赖锁、模型清单、workflow、配置、启动命令和证据目录，形成不可变部署制品。
6. 使用同一制品完成目标环境启动和 smoke；随后实际执行回滚演练。已有批准版本时验证其可启动并完成代表性 workflow；首次部署时验证能恢复预先定义的部署前状态或 N2 干净基线。
7. 根据本节点结果给出批准或拒绝结论；任何绑定字段变化都按第 6 节返回相应节点重新验收。

#### 需要查询和填写

- smoke、稳定性和性能用例。
- 冷/热启动、重启、负载、并发、持续时间、预热和采样要求。
- 成功率、延迟、吞吐、质量和资源阈值。
- 日志、指标、告警、磁盘、缓存和临时文件监控。
- 服务账户、监听范围、访问控制和自动启动。
- 最终制品、上一批准版本、备份和回滚目标。
- 首次部署没有上一批准版本时，部署前状态、干净基线和卸载/停用恢复目标。

#### 必须校验

##### Smoke

- 从停止状态完成冷启动和健康检查。
- 枚举目标设备、节点和模型。
- 执行最小代表性 workflow 并生成有效输出。
- 受控停止、再次启动和再次推理通过。
- 无残留进程、端口、队列和锁文件。

##### 稳定性与性能

- 按预先批准的次数、并发和持续时间执行。
- 覆盖首次、预热、连续和并发场景。
- 统计成功率、延迟、吞吐、质量和峰值资源。
- 内存、句柄、线程、缓存、日志和临时文件无异常持续增长。
- 无崩溃、死锁、设备重置和未批准 fallback。
- 阈值必须测试前批准，不能完成后倒推。

##### 生产晋级

- N0-N4 和本节点测试全部通过。
- 冻结 deployment manifest、依赖、workflow、模型和配置摘要。
- 生产使用同一已验收制品，不重新构建浮动版本。
- 最小权限、网络暴露、访问控制、轮转、容量和告警通过。
- 生产启动后重复同一 smoke。

##### 回滚

- 如果存在上一批准部署，其制品、配置、模型和 workflow 必须可取得并能完成代表性 workflow。
- 首次部署没有上一批准制品时，回滚目标定义为：停止并移除新服务绑定，恢复部署前系统状态，或恢复到已验收的 N2 干净基线；具体选择必须在测试前写入记录，不能留空。
- 回滚不依赖浮动地址重新下载。
- 回滚触发条件、责任人、步骤和目标时间明确。
- 实际演练回滚并验证健康检查和代表性 workflow。
- 不可逆迁移具有兼容备份或前向修复方案。

#### 完成条件与证据

- smoke、稳定性、性能、生产检查和回滚满足批准标准。
- 非生产部署可将生产晋级标记为获批 `N/A`，但 smoke、稳定性和回滚不能省略。
- 证据：运行记录、趋势、输出、监控、晋级审批、最终 manifest 和回滚演练。

#### 常见问题与建议

| 问题现象 | 可能原因 | 解决建议 | 重新验收条件 |
|---|---|---|---|
| 冷启动失败，手工启动成功 | 账户、目录或环境不同 | 将绝对路径和服务环境纳入配置 | 冷启动/重启/smoke 重验 |
| 首次任务很慢 | 编译、加载或预热 | 分离冷/热指标，定义预热流程 | 冷启动和热性能均通过 |
| 连续运行 OOM/崩溃 | 泄漏或缓存无界 | 定位节点/workflow，限制缓存或换批准版本 | N3/N4 和稳定性重验 |
| 并发错误率升高 | 队列或资源竞争 | 设置验证过的并发、队列和背压 | 目标并发重验 |
| 日志/输出填满磁盘 | 无轮转和保留策略 | 配置轮转、配额、清理和告警 | 持续运行与告警重验 |
| 生产与验收性能不同 | 制品、配置或设备漂移 | 使用同一制品，核对 manifest | 生产 smoke/性能重验 |
| 停止后任务仍运行 | 取消未传播到队列/子进程 | 验证 interrupt、队列清理和进程终止 | 停止、重启和资源释放通过 |
| 回滚后不能启动 | 回滚包不完整或格式不兼容 | 使用并行版本目录和兼容备份 | 回滚演练完全通过 |
| 告警未触发 | 指标、规则或通知链错误 | 注入受控故障验证链路 | 告警演练通过 |

## 11. 官方来源维护

执行 N1/N2 时查询并保存执行日最新的：

- ComfyUI 官方系统要求、安装说明、release notes 和资产元数据。
